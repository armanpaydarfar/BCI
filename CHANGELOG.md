- **`tools/deploy_robot_gaze.sh` single-command wrapper.** Invokes the C++ repo's `build_and_deploy.sh --push 192.168.2.1 --tool Gaze_Tracking` after checking the cross-repo `deploy.config` and the built `dist/01d91ea/Gaze_Tracking` exist. - **Single cross-platform superset env + role-by-steps + repo hygiene.** There is now **one** env file, `environment.yml` (conda env `lsl`, Python 3.12): a curated, cross-platform **superset** installed identically on every host, so every module imports on every unit. "Role" (`tools/bootstrap_machine.sh --role`) selects *which setup steps run*, not a different dependency set — `control` (Linux operator host) runs the full setup; `server` (GPU perception host, Linux or Windows) skips the control-only steps. No torch knob is needed: the default PyPI torch wheel is already a CUDA build (validated as `2.12.0+cu130`, `cuda.is_available()` True) that also runs on CPU, so a plain create works on both GPU servers and CPU-only hosts; `--role server` just verifies CUDA is visible (a CPU-only box can optionally install the `.../whl/cpu` wheel to skip the CUDA download). The file is a **curated spec, not a frozen export** — conda-forge-only with no OS-pinned packages — so the same file create-solves on Linux and native Windows; only the numerical/decoder core (`numpy`/`scipy`/`scikit-learn`/`pyriemann`/`python`) is version-pinned, everything else floats to the per-OS solve (with a `pandas<3` range guard so the 2.x analysis scripts don't hit the 3.0 breaking major). This **retires `environment.server.yml` and the `harmony-server` env** (replaced the earlier control-superset / server-subset split, which was never released): the old `environment.yml` was a fully-pinned linux-64 export that could not solve on Windows. - **`vlm_service.py` exposes `--max-output-tokens` (default `8192`)** and passes it through to `IntentReasoner(max_tokens=...)`. Billing is per emitted token, not per cap. Override via `--max-output-tokens` or `VLM_MAX_OUTPUT_TOKENS` in config.
